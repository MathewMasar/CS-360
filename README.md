# CS-360

The goal of the Inventory Management App was to provide users with a simple, mobile-friendly way to track products, monitor quantities, and receive alerts when inventory reached zero. The primary user need addressed was real-time awareness of stock levels, combined with an intuitive interface for adding, editing, and deleting products. The app was designed for individuals and small-scale users who require a lightweight inventory solution without unnecessary complexity.

To support these needs, several key screens were necessary: a login screen, an inventory grid screen, product detail view, add/edit product forms, and an SMS alerts screen. The inventory screen used a two-column grid layout to provide clear visual organization and reduce scrolling. Forms were designed to show only essential fields to minimize cognitive load. Alerts were displayed in a dedicated section with clear grouping and dismiss functionality, ensuring users could manage notifications easily. These design decisions kept user efficiency and clarity in mind, resulting in a clean, user-centered interface that supported the primary workflow of viewing and managing inventory.

When coding the app, I approached development in structured stages. I first defined the necessary data models, such as the Product object, and then built the database schema to store and retrieve those objects. I worked screen by screen, implementing features incrementally and testing after each addition. I also ensured that UI elements properly received formatted data objects from the database and that state changes were reflected in RecyclerView updates. This structured, iterative strategy can be applied in the future to reduce debugging time and ensure system cohesion as complexity increases.

Testing was done through repeated builds, clean installs, permission validation, and full user-flow walkthroughs from login to alert generation. I tested edge cases such as duplicate product IDs, quantity changes triggering alerts, and permission denial scenarios. This process was essential because it revealed integration issues between activities, database updates, and UI refresh logic that would not have been apparent by testing features in isolation. End-to-end testing ensured the entire system functioned cohesively.

Throughout the design process, one of the most innovative solutions involved developing a persistent alert system that dynamically regenerated out-of-stock alerts while allowing users to dismiss them temporarily. This required integrating database checks, object state updates, and UI refresh logic in a coordinated way. It demonstrated system-level thinking rather than simple feature coding.

I was particularly successful in implementing the database-driven alert lifecycle system. This component required managing object modeling, database schema design, runtime permission handling for SMS, RecyclerView updates, and UI feedback in a unified workflow. Successfully integrating these elements demonstrated growth in my understanding of full-cycle mobile application development and system architecture.
